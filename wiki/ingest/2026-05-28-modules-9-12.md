## Module 9 — AI Invocation Logging

Created `packages/workers/src/log-ai-invocation.ts`:
- `logAIInvocation()` helper writes to the partitioned `ai_invocations` table; errors are swallowed so logging never blocks the pipeline.
- `estimateCostMicro(model, inputTokens, outputTokens)` computes micro-dollar cost estimates for common models.
- Wired into all 4 active AI callsites: `classify.ts` (reply_classification), `draft.ts` (draft_generation), `signal-extraction.ts` (signal_extraction), `analytics-intelligence.ts` (analytics_intelligence).
- Each callsite logs on both success and failure paths with `traceId`, `entityType`, `entityId`, `latencyMs`, `validatedOutput`, and `costUsdMicro`.

## Module 10 — Sequence Generation Pipeline

**Migration** `20260528000003_create_generated_sequences.sql`:
- `generated_sequences` table: `client_id`, `campaign_id`, `lead_id`, `emails JSONB`, `strategy_notes`, generation metadata, push state (`pending → pushing → pushed/failed`), `instantly_campaign_id`, `instantly_contact_id`, `pushed_at`.
- Index on `status` for pending/pushing jobs.

**Migration** `20260528000004_add_instantly_campaign_to_clients.sql`:
- Adds `instantly_campaign_id TEXT` column to `clients`.

**New client** `packages/workers/src/clients/instantly-outbound.ts`:
- `addLeadToInstantlyCampaign()` — POST to `api.instantly.ai/api/v1/lead/add` with `skip_if_in_workspace: true` and per-step email customisation.
- `getInstantlyCampaign()` — fetch campaign status.

**New worker** `packages/workers/src/workers/sequence-generation.ts` (AI invocation point 2):
- Loads lead + enriched_lead + client context.
- Calls `provider.generateSequence()` with ICP signals and personalization hooks.
- Logs AI invocation, writes result to `generated_sequences`, updates lead status to `campaign_ready`.
- Enqueues `instantly-push` job with dedup `jobId: seq-gen:<clientId>:<leadId>`.
- Emits `sequence_generated` event.

**New worker** `packages/workers/src/workers/instantly-push.ts`:
- Reads `generated_sequences` row, calls `addLeadToInstantlyCampaign()` with per-step emails.
- On success: marks sequence `pushed`, writes `instantly_contact_id`, updates lead to `queued_for_sending`.
- On failure: marks sequence `failed`, writes `push_error`, re-throws for BullMQ retry.
- Emits `campaign_pushed` event.

**New queues**: `sequenceGenerationQueue` ("lead-sequence-generation"), `instantlyPushQueue` ("lead-instantly-push") — added to `queues.ts` and `public.ts`.

**Dashboard route** `apps/dashboard/app/api/sequences/generate/route.ts`:
- POST `{ clientId, leadId, campaignId? }` → enqueues sequence_generation with dedup jobId, returns 202.

## Module 11 — Objection Intelligence Worker

**New queue**: `objectionIntelligenceQueue` ("reply-objection-intelligence") — added to `queues.ts` and `public.ts`.

**New worker** `packages/workers/src/workers/objection-intelligence.ts` (AI invocation point 5):
- Triggered by `classify.ts` when `intent === "OBJECTION"` — runs in parallel with draft generation.
- Loads raw reply body, prior conversation history (last 6 exchanges), client context.
- Calls `provider.analyzeObjection()` — returns `objection_category`, `severity`, `recommended_approach`, `response_draft`, `escalate`, `escalation_reason`.
- Writes enriched `objection_type` and analysis notes back to `reply_classifications`.
- If `escalate: true`, updates `reply_items.status = 'ESCALATED'`.
- Logs AI invocation, emits `objection_analyzed` event.

**Update** `classify.ts`: enqueues `objection-intelligence` job immediately after classification when intent is OBJECTION.

## Module 12 — Feature Flags & Config Runtime

**New helper** `packages/workers/src/config.ts`:
- `getFeatureFlag(clientId, featureKey)` — queries `feature_flags` table with 60s Redis cache; defaults to `true` on missing row.
- `getGlobalConfig<T>(configKey)` — queries `global_config` table with 120s Redis cache; returns `null` if not found.
- Both helpers are resilient: Redis errors fall through to DB; DB errors bubble up.

**Migration** `20260528000005_seed_booking_reminders_flag.sql`:
- Seeds `booking_reminders` feature flag (enabled: true) — was missing from the original seed.

**Wired into workers:**
- `crm-sync.ts` — checks `crm_sync` flag before proceeding; returns `skipped` if disabled.
- `booking-reminder.ts` — checks `booking_reminders` flag before sending; returns `skipped` if disabled.
- `analytics-aggregator.ts` — checks `analytics` flag per client; skips clients with analytics disabled.
- `send.ts` — reads `sending_limits` global config to get `concurrent_sending_jobs` limit.
